# Phase 6.5 — Artifact ingestion *(re-runnable)*

> Ingest a user-provided artifact — a deployed site, a Figma file, a one-shot AI-generated landing page, a partial Framer / Webflow / Wix / WordPress project, a freelancer delivery, or the round-trip output of the JSON handoff protocol — and integrate it into project state without losing prior work. **The only re-runnable phase. Can fire at any project lifecycle point.**

## Mission

The website-builder picks users up wherever they are. Most users do not arrive greenfield — they arrive with *something*: a site they already deployed, a Figma file a designer handed them, a one-shot landing page ChatGPT or v0 or Lovable produced, a half-built Framer project, a freelancer's HTML delivery. Phase 6.5 is the mechanism that takes that artifact and folds it into structured project state — design tokens into `brand.yaml`, content into `content/pages/{slug}.md` + `content/strings/{lang}.json`, component shapes into `components.yaml`, with a versioned audit trail in `decisions/ingest-{timestamp}.md`.

Phase 6.5 is unique in the pipeline in two ways. First, it is **re-runnable**: it fires at session-start when the entry mode is non-greenfield, *and* it fires any time during the project lifecycle when the user invokes it (the canonical case: a user at phase 18 generates a new section in ChatGPT and wants it integrated without restarting). Second, its **defining discipline is conflict protection**: every conflict between incoming artifact state and existing project state halts and forces an explicit user decision. No silent merge. No silent overwrite. Protecting the user's prior work matters more than completion speed.

The MVP extraction tool is **Stitch** (per locked decision 55). The phase 6.5 contract leads with Stitch as the primary URL/site-extraction path. divmagic, the Figma design-to-json plugin, the AI-output parser, Playwright walks, and the JSON-handoff-output ingestor are documented as expansion paths and entry-mode-specific paths — the AI-output parser and JSON-handoff ingestor are in fact load-bearing in MVP (they handle the `has-ai-output` entry mode and the phase-18 round-trip), so they are first-class here; divmagic + Figma-plugin + Peel/Codia are catalogued expansion paths beyond MVP.

## Entry conditions

Phase 6.5 fires when **any** of these is true (it does NOT have a strict single predecessor — `prev_phase: 6` represents its natural slot in the entry-time route, not a hard gate):

1. **Session start with a non-greenfield entry mode.** `project.yaml.entry_mode` is one of `has-existing-site` / `has-ai-output` / `has-Framer-attempt` / `has-Figma-file`, and the artifact has not yet been ingested. In this case 6.5 runs after phase 6 (wild content capture) in the entry-time route — the artifact is captured to `inbox/` during phase 6, then 6.5 structures it.
2. **User explicit invocation.** The user says "ingest this", "I have a new artifact", or pastes a URL / Figma export / HTML / code at any point.
3. **Detected file in `.website-builder/outputs/`.** The agent finds a pasted-back external-tool output (typically from the JSON handoff protocol round-trip) and surfaces ingestion as the next action.
4. **Phase 18 handoff round-trip.** During phase 18 (component build) the agent emitted a request brief, the user generated the component externally (ChatGPT / v0 / Cursor / human freelancer), and pastes the result back — 6.5 ingests it and returns control to phase 18.

Because 6.5 is re-runnable, `current_phase` is **not** advanced to 6.5 as a linear step. Instead, the agent records the *return phase* (where the project was before 6.5 fired) and resumes there after ingestion. `next_phase: 7` represents only the entry-time route (after the entry-mode artifact is ingested during onboarding, the project proceeds to phase 7 asset audit). For mid-project re-runs the agent returns to the recorded return phase.

## What Claude must establish

The work product is updated project state + a versioned ingestion decision log. Concretely the agent runs the 8-step phase 6.5 flow (from `DESIGN-ingestion-and-extraction.md`):

1. **Identify artifact type + select extractor.**
   - URL / deployed site → **Stitch** (primary, per decision 55), optionally paired with **Playwright walk** for dynamic state
   - Figma file → **Figma design-to-json plugin** *(expansion path; MVP-documented but not MVP-primary)*
   - HTML / React / Vue / Svelte source code → **AI-output parser** (load-bearing in MVP for `has-ai-output` + phase-18 round-trip)
   - Screenshot → **Stitch** (vibe extraction) or Tailwind Theme Maker *(expansion)*
   - Live auth-walled site → **Playwright walker** with user-supplied auth
   - JSON-handoff-protocol output → **AI-output parser** via the handoff ingestion contract
   - Element-precision extract → **divmagic API** *(expansion path beyond MVP)*

2. **Run extractor; capture raw output.** Stitch → a design spec (the portable DESIGN.md / vibe-spec format Stitch emits); AI-output parser → AST + extracted strings + extracted classes; Figma plugin → structured JSON; Playwright → DOM walk + per-section screenshots.

3. **Normalize into project-state shapes.** Design tokens → `brand.yaml` (Layer 1); page structure → `sitemap.yaml` (Layer 2); section / component shapes → `components.yaml` (Layer 2); microcopy → `content/strings/{lang}.json` (Layer 3); page prose → `content/pages/{slug}.md` (Layer 4); iteration history → `briefs/{component}-{ts}.json` (Layer 5).

4. **Detect conflicts** with existing project state — token conflict (incoming primary ≠ existing primary), component conflict (incoming Hero shape ≠ existing Hero shape), string conflict (incoming "subscribe" copy ≠ existing), page conflict (incoming `/about` ≠ existing `/about`).

5. **Surface every conflict; collect explicit user decisions.** For each conflict the agent uses `AskUserQuestion` with the options: keep current / use incoming / merge (with explicit reasoning). Per locked decision 36 the default is **halt + force user decision** — no auto-resolution even when the incoming value seems obviously better.

6. **Apply non-conflicting changes + user-resolved conflicts** to the relevant state files.

7. **Re-run cross-layer validation** (same checks as the session-start hook: all YAML/JSON/MD parses, frontmatter conforms, `{strings.x.y.z}` references resolve, sitemap pages exist in `content/pages/`, sections exist in `components.yaml`, token refs resolve).

8. **Determine next phase.** If the project's prior-phase exit criteria still hold → resume the recorded return phase. If new content surfaces gaps → route to the relevant phase. If the user wants to keep going where they were → resume there with new content available.

## Gating rules

Phase 6.5's gating discipline is the inverse of most phases: it does not gate the user from *advancing*; it gates the agent from *silently mutating state*. The non-negotiable rule (locked decision 36):

1. **No silent overwrite, ever.** Every conflict between incoming and existing state halts the ingestion and surfaces the conflict to the user via `AskUserQuestion`. The agent refuses to apply an incoming value over an existing value — even an "obviously better" one — without an explicit user decision. The agent surfaces the *cost* of each option (e.g., "accepting incoming primary means phase 17 review re-runs on the 4 components already built against the old primary").

2. **No silent merge.** When the agent could "intelligently" merge two component shapes or two page versions, it still refuses to do so silently. Merge is offered as an explicit option with the agent showing both sides; the user chooses merge knowingly.

3. **No partial ingestion masquerading as complete.** When the artifact is incomplete (broken HTML, partial Figma export, truncated paste), the agent refuses to report ingestion as successful. It surfaces the gap: "the pasted code references `<PricingCard />` but the component definition isn't in the paste — what's missing?" and asks the user to supply the missing piece or explicitly accept a partial ingestion logged as such.

4. **No state mutation without a decision log.** Every 6.5 run produces a `decisions/ingest-{timestamp}.md` even when there were zero conflicts. The audit trail is non-optional; re-runnability is only safe because every run is recorded and reversible.

5. **No phase advancement masked as ingestion.** Phase 6.5 does NOT advance `current_phase` as if it were a linear step. It records `return_phase` and resumes there. The agent refuses to "skip ahead" by way of an ingestion — if the artifact implies the project is further along than it is (e.g., a fully-built site ingested at phase 6), the agent surfaces that and asks whether to backfill the intermediate phases' state or treat the artifact as reference.

There is no `## Skip authorization` section for this phase. **Phase 6.5 is re-runnable, not skippable** — it does not sit in the linear pipeline as a step that could be skipped; it is a side-channel that fires on demand. There is no "skip phase 6.5" decision to authorize; if a user has no artifact, phase 6.5 simply never fires. This omission is deliberate per the schema's optional-section rule.

The override path within 6.5: the user can explicitly choose to accept a partial ingestion, accept incoming-over-existing for a conflict, or treat an over-complete artifact as reference-only — every override is logged in the ingestion decision file with the user's reasoning.

## Tools and skills used

- **Stitch** *(primary extraction tool — MVP per decision 55)*. As of the current capability check (context7 `/google-labs-code/stitch-sdk` + `/llmstxt/stitch_withgoogle_llms_txt`, freshness 2026-05-18), Stitch's primary mode is vibe-driven UI generation + adjective-driven styling + sketch-to-UI, with HTML + screenshot extraction. Two invocation paths:
  - **Browser-in-loop (MVP-primary, matches `DESIGN-extraction-stitch.md`):** the agent walks the user through running Stitch at https://stitch.withgoogle.com against their URL / screenshot, the user copies the resulting design spec back, the agent normalizes it. This is the documented MVP path and the path the contract leads with.
  - **SDK / CLI / MCP (evolution path, use when present):** a `@google/stitch-sdk` (env `STITCH_API_KEY`, `project.generate()` → `screen.getHtml()` / `screen.getImage()`), a Gemini-CLI Stitch extension, and a Stitch MCP now exist. When the user has the SDK key or MCP configured, the agent uses the programmatic path (no user-in-the-loop). The agent checks for `STITCH_API_KEY` / the Stitch MCP at 6.5 entry; absence → browser-in-loop; presence → programmatic. *(This evolution was confirmed via fresh context7 lookup; the design doc `DESIGN-extraction-stitch.md` predates the SDK and documents browser-only — the contract honors the design-doc MVP framing as primary while surfacing the evolved path.)*
- **AI-output parser** *(load-bearing in MVP — handles `has-ai-output` entry mode + phase-18 handoff round-trip)*. Built into the plugin; no external service. Reasoning + light AST parsing extracts design tokens (from inline styles + Tailwind/shadcn/CSS classes), content/strings (page prose vs reusable strings), component shapes (JSX/template structure), external dependencies (import statements). Implemented per `${CLAUDE_PLUGIN_ROOT}/extraction/ai-output.md`.
- **`mcp__playwright__*`** — the Playwright walker. Drives a real browser to walk a deployed site, capture per-section screenshots, exercise hover / scroll-triggered / mobile-emulation state Stitch's static analysis misses. Used for `has-existing-site` mode when the site has dynamic state, and for auth-walled sites (user supplies credentials; Playwright authenticates; walks the authenticated state). Implemented per `${CLAUDE_PLUGIN_ROOT}/extraction/playwright-walk.md`.
- **`AskUserQuestion`** — the conflict-resolution tool. Every detected conflict surfaces via `AskUserQuestion` with keep-current / use-incoming / merge options + per-option cost. Heavy use; conflict resolution is the defining interaction of this phase.
- **`Read` / `Write` / `Edit`** — read existing project state to detect conflicts; write/edit `brand.yaml`, `sitemap.yaml`, `components.yaml`, `content/pages/{slug}.md`, `content/strings/{lang}.json`, `briefs/{component}-{ts}.json`; author the `decisions/ingest-{timestamp}.md` log.
- **`Bash`** — for artifact handling: scrape a deployed URL into `inbox/`, `curl` divmagic when used, `ls` / `file` to inspect pasted-back files in `outputs/`, run the cross-layer validation script.
- **`WebFetch`** — to scrape a deployed-site URL when Stitch browser-in-loop isn't applicable (small static site the agent can read directly) or to fetch a public Figma-community-plugin's expected JSON schema for normalization.

**Expansion-path extractors (catalogued, beyond MVP per decision 55):**

- **divmagic API** — element-precision extraction via HTTP (Bash + curl; API key in `keys.yaml` if user has one). Used when the user wants a specific element extracted rather than a whole design system. Catalogued in `DESIGN-ecosystem-catalog.md`; expansion phase 10.
- **Figma design-to-json plugin** — user runs the community plugin in Figma, pastes JSON; agent normalizes. Handles `has-Figma-file` entry mode. MVP-documented (the entry mode exists in MVP) but the plugin integration is an expansion path; in MVP the agent can also handle Figma files via Stitch screenshot extraction as a fallback.
- **Peel Studio / Codia VisualStruct / Tailwind Theme Maker** — additional screenshot/URL extractors catalogued in `DESIGN-extraction-peel-codia-others.md`; expansion phase 10; surfaced only when Stitch fails on a given artifact.

No subagent spawn by default. When multiple extractors must run on one artifact (Stitch for tokens + Playwright for dynamic content + AI-output parser for component shapes), the agent orchestrates them sequentially in-person; it may spawn a research subagent only for genuinely parallelizable multi-source extraction.

## Output artifacts

| Path | Schema | Purpose |
|---|---|---|
| `.website-builder/decisions/ingest-{timestamp}.md` | Ingestion decision log per schema above | **Always produced** (even zero-conflict runs); the audit trail that makes re-runnability safe + reversible |
| `.website-builder/brand.yaml` | Layer-1 tokens updated/merged | Design tokens extracted from the artifact (or seeded if greenfield-with-artifact) |
| `.website-builder/sitemap.yaml` | Layer-2 page structure updated | Page hierarchy extracted from a deployed site / multi-page artifact |
| `.website-builder/components.yaml` | Layer-2 component shapes updated | Component shapes extracted from code/Figma artifact |
| `.website-builder/content/pages/{slug}.md` | Layer-4 page prose updated/created | Page prose extracted from the artifact |
| `.website-builder/content/strings/{lang}.json` | Layer-3 CDJSON updated | Microcopy extracted from the artifact |
| `.website-builder/briefs/{component}-{ts}.json` | Layer-5 iteration history updated | When the artifact is a handoff round-trip, iteration history is appended so the next round has context |
| `.website-builder/inbox/{...}` | Raw artifact preserved | The original artifact is preserved in `inbox/` before normalization (provenance) |

The ingestion decision log is the one always-required artifact. Which state files change depends entirely on what the artifact contained + how conflicts resolved.

The agent does NOT set `current_phase` to a new linear value as a result of 6.5. It records `return_phase` in the decision log and resumes the project there. For the entry-time route (non-greenfield onboarding), after ingestion the agent advances to phase 7 (asset audit) — that is `next_phase: 7`.

## Common failure modes

**The user pastes incomplete code.** Broken JSX, missing imports, a component referencing an undefined child. The agent surfaces the gap precisely: *"The pasted code renders `<PricingCard />` three times but there's no `PricingCard` definition in the paste. Three options: (a) paste the missing component too; (b) I infer a `PricingCard` shape from how it's used and you confirm; (c) ingest the rest now and log `PricingCard` as a known gap to resolve before phase 19. Which?"* Never reports success on a partial ingestion.

**Token conflict — incoming primary differs from existing.** Incoming `oklch(58% 0.16 240)`, existing `oklch(60% 0.18 245)`, and phase 17 was already complete with 4 components built against the existing value. The agent halts + surfaces with full cost: *"The artifact's primary color is slightly different from your locked primary. Existing: oklch(60% 0.18 245), and phase 18 already shipped 4 components referencing it via CSS variables. Three options: (a) keep existing — recommended, the 4 components stay valid, the incoming artifact's color gets remapped to your token; (b) accept incoming — phase 17 review re-runs and the 4 components need a token-reference re-check; (c) blend to an intermediate value with explicit reasoning — both existing components and incoming artifact get adjusted. Which?"* Decision logged with reasoning.

**Component shape conflict.** Incoming `Hero` declares `{title, subtitle, cta_text, cta_href}`; existing `components.yaml.Hero` declares `{headline, body, cta}`. The agent surfaces: *"Component shape conflict on Hero. (a) keep existing shape — the incoming artifact's Hero gets re-mapped to your prop names; (b) accept incoming shape — your existing Hero code at [path] needs prop renames; (c) audit both prop-by-prop and decide each. Which?"* Per-prop audit option exists for cases where the shapes are partly compatible.

**Page conflict — incoming `/about` differs from a recently-edited existing `/about`.** The agent surfaces with recency awareness: *"Conflict on `about.md`: yours has 412 chars edited 12 minutes ago; incoming has 1,847 chars from the artifact. (a) keep yours; (b) replace with incoming; (c) merge — I'll show you both side-by-side and you pick per-section; (d) save incoming as `about-v2.md` and decide later. Which?"* Recent edits get extra weight in how the option is framed (the agent surfaces "you just edited this 12 minutes ago" rather than burying it).

**The artifact is in a different language than the project default.** Incoming artifact is German; project default is English. The agent does not treat German strings as English: *"This artifact is in German; your project default is English. I'll treat the incoming content as DE — strings go to `content/strings/de.json`, page prose gets `language: de`. You'll need EN versions; phase 16's translation pattern handles that (per DESIGN-i18n.md Pattern 1 inline, or Pattern 2 translator handoff). Proceed?"*

**The artifact's stack doesn't match the project stack.** User pastes React but the project stack (set at phase 11) is Astro. The agent surfaces: *"This is React; your project stack is Astro. (a) I translate the component to Astro (straightforward for this component — mostly markup + props); (b) you wanted to switch stacks (that's a phase-11 re-decision with downstream consequences); (c) ingest the design intent only (tokens + content + shape) and rebuild in Astro at phase 18. For a simple component (a) is usually right; for a complex stateful one (c) is safer. Which?"*

**Brand drift — incoming colors look nothing like the existing brand.** The agent surfaces the drift as a *question of intent*, not an error: *"The incoming artifact's palette is very different from your locked brand (existing is warm-orange-led; incoming is cool-blue-led). Is this an intentional rebrand, or did the artifact's source (e.g., a default ChatGPT template) just not follow your brand? If intentional rebrand — we re-run phase 17 review. If template-default — we keep your brand and remap. Which?"*

**Hidden dependencies — incoming code imports libraries the user doesn't have.** The agent surfaces per `tool-dependency-discipline.md`: *"This code imports `framer-motion` and `@radix-ui/react-dialog` — neither is in your project. (a) add them as dependencies (I'll verify they're the real packages, not typo-squats, and add to your stack's manifest); (b) rewrite the component without them (more work, fewer deps); (c) note the gap and resolve at phase 18. Which?"* The agent never blindly installs; it surfaces the dependency decision.

**Mom's canonical pattern — mid-project new-section ingestion at phase 18.** The user is at phase 18 (component build), has an idea for a Testimonials section, generates it in ChatGPT, pastes the output. The agent: ingests via the AI-output parser → extracts the Testimonials component shape + the placeholder copy + any inline tokens → detects conflicts (the ChatGPT output used its own blue; the project primary is locked orange) → surfaces the token conflict (keep existing orange, remap) → integrates the new `Testimonials` component into `components.yaml`, the section into the relevant page's `content/pages/{slug}.md`, the copy into `content/strings/{lang}.json` → logs `decisions/ingest-{ts}.md` with `return_phase: 18` → resumes phase 18 with the new component available. **No prior work lost. The project did not restart.** This is the load-bearing use case the whole re-runnability design exists for.

**The user re-runs 6.5 against the same URL twice.** Stitch extraction re-run produces a new spec. The agent surfaces: *"You ran ingestion on this URL before (ingest-2026-05-15...). Re-running will produce a fresh extraction. (a) treat as update — diff against the last ingestion, surface only what changed; (b) treat as fresh — full conflict pass against current project state; (c) cancel. Which?"* Re-runs are first-class; the prior decision log makes the diff possible.

**The user pastes a Stitch vibe-prompt instead of Stitch output.** Because Stitch's Agent Protocol directs the *agent* to formulate a vibe-prompt for the *user* to paste into Stitch, a confused user may paste the prompt back instead of the generated result. The agent detects this: *"That looks like the vibe-prompt I'd send you to paste *into* Stitch, not Stitch's output. The flow is: I give you a vibe-prompt → you paste it into stitch.withgoogle.com → Stitch generates a design → you copy *that* back to me. Want me to regenerate the vibe-prompt and walk through it?"*

**Stitch SDK / MCP is present but mis-keyed.** If `STITCH_API_KEY` is set but invalid, the agent surfaces per `tool-dependency-discipline.md` Tier 2 (third-party tool): *"The Stitch SDK key is configured but the API rejected it (401). (a) re-check the key (per DESIGN-secrets-and-keys.md key rotation); (b) fall back to browser-in-loop Stitch (you run it manually at stitch.withgoogle.com); (c) skip Stitch and use the AI-output parser if the artifact is code. Which?"* Falls back to the documented MVP browser path; never silently degrades.
