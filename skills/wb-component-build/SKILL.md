# wb-component-build — Phase 18: Component design / build

> The phase where the design system becomes code. For **every** component named across phases 13–15, produce a spec in `components.yaml` and working code in the user's chosen stack, written against the phase-17 tokens and reviewed by the user. The single heaviest phase in the pipeline and the most context7-dependent — component-library and framework APIs drift fast.
>
> Work from the design docs directly. The substantive references are the phase contract and the per-ecosystem component design docs (paths in `## Design-doc sources`). This skill is the procedural layer; the design docs are the source of truth for vocabulary, selection logic, and decisions.

## Phase pointer

This skill drives exactly one pipeline phase:

- **Phase 18 — Component design / build.** Read `phase-contracts/18-component-design-build.md` (in the plugin root) **verbatim at phase entry**. It carries the authoritative Mission / Entry conditions / What-Claude-must-establish / Gating rules / Tools / Output artifacts / Common-failure-modes / context7-freshness for this phase. This skill tells the agent *how* to execute that contract; the contract tells the agent *what the contract is*.

Phase 18 is `re_runnable: false` for the initial build, but re-entered on a component-library switch or when the JSON handoff protocol returns external output (which flows through phase 6.5, then back here for integration).

## What this phase produces

Two always-required artifacts plus three conditional ones:

| Artifact | Always? | Shape |
|---|---|---|
| `.website-builder/components.yaml` | yes | spec (props/states/responsive/a11y/tokens_used) + impl (library/composes/file/reviewed_by_user) per component — schema in the phase contract |
| Component code in the user's project | yes | per chosen-stack convention (e.g. `components/ui/*.tsx`, `code/*.tsx`, custom Gutenberg blocks) |
| `.website-builder/project.yaml` `component_library` | yes | `{ primary: <lib>, complementary: [<lib>] }` — the locked choice, enforced through phases 19–23 |
| `.website-builder/briefs/{component}-{ts}.json` | only if handoff used | `component-request-v1` shape |
| `.website-builder/outputs/{component}-{ts}.{ext}` | only if handoff used | whatever the external tool returned |
| `.website-builder/library/docs/{library}.md` | auto-cloned at entry | current chosen-library docs via context7 |

## Procedure

### 1. Verify entry conditions, then read the contract

Refuse to start if any is unmet (the contract enumerates these — surface the specific gap, do not paper over):

- Phase 17 complete — `brand.yaml.tokens` + `brand.yaml.tokens.css` exist (OKLCH colors, type scale, spacing, motion, dark-mode). A component that does not reference these is a gating failure.
- Phase 11 complete — `project.yaml.stack` + `project.yaml.transactional` set. The stack determines the component-library candidate set.
- Phase 9 complete — `sitemap.yaml` exists and is locked. Phases 13–15 derive the component inventory from it; this is the transitive prerequisite for a complete inventory.
- Phases 13–15 complete — `content/pages/*.md` + `content/sections.yaml` name every component the site needs. This inventory is the contract phase 18 must fully satisfy.
- Phase 16 complete — components are built against real content, not lorem ipsum.

Read `phase-contracts/18-component-design-build.md` verbatim before proceeding.

### 2. Run context7 at phase entry — non-negotiable

This is the most context7-dependent phase. Training data is stale on every fast-moving library here (shadcn moved to Tailwind v4 + React 19; Radix maintenance status shifted; Base UI's package name changed across releases; NextUI is rebranding to HeroUI; Chakra v2→v3). Per `cross-cutting/DESIGN-context7-integration.md` and `.claude/rules/context7.md`:

1. `mcp__context7__resolve-library-id` then `mcp__context7__query-docs` for the chosen component library — verify the **current** install command, theming-integration shape, primitive-composition pattern, and package name *before writing a line*.
2. Same for the framework (Next.js App Router Server/Client rules; Astro island directives; Payload v3 RSC; Framer Code Components API).
3. Cache into `.website-builder/library/docs/{library}.md` (clone-into-project, decision 42) — phase 18 queries the same library dozens of times; read the local cache first, re-fetch on freshness flag.
4. On context7 unreachable: Tier-2 per `.claude/rules/tool-dependency-discipline.md` — surface explicitly, fall back to WebFetch of the canonical docs URL, log the gap; never silently use training data for a named library's API.

For per-stack context7 library-id mappings, read `references/per-stack-codegen.md`.

### 3. Select the component library — surface options, user picks

Component-library selection happens at this phase. **Always surface the default + 1–2 alternatives with trade-offs via `AskUserQuestion`; the user picks.** The full selection logic per stack-class lives in `references/component-library-selection.md` — read it. Headlines:

- React stack, no strong preference → **shadcn/ui** primary (copy-paste, user owns code, Tailwind-v4 + OKLCH aligns exactly with phase-17 output). Propose **Magic UI** for motion-heavy sections, **Aceternity UI** if phase-2 vision leaned premium/agency/portfolio.
- "Everything out of the box" → **Mantine**. "Rapid prototyping" → **Chakra UI**. Admin/dense-data → **Ant Design** / **Material UI**. Consumer polish + motion → **NextUI/HeroUI** (verify naming via context7 — rebrand in flight).
- Non-React Tailwind stacks (Hugo / Astro-pure / static-HTML / WordPress-Tailwind) → **DaisyUI**; pair with **Headless UI** / **Alpine.js** for JS interactivity DaisyUI's CSS-only patterns don't cover. Surface **Tailwind UI**'s $299 cost explicitly if proposed.
- Vue → **PrimeVue** (Aura theme) default; **Vuetify** for Material; **Element Plus** for dense data. Single-library is the Vue norm.
- Svelte → **Skeleton UI** (comprehensive) or **Bits UI / shadcn-svelte** (source-ownership); **Melt UI** for total control.
- Total control / missing primitive / design system from scratch → **headless primitives**. Verify current maintenance status via context7: as of 2026-05 Radix update velocity slowed (WorkOS-acquired) and Base UI (MUI-team) is the more actively-maintained primitive layer — surface this rather than baking either as canonical; shadcn/ui now has a Base UI path.

Record the choice: `project.yaml.component_library: { primary: <lib>, complementary: [<lib>] }`.

### 4. Build each component — default is write the code

Per locked decision 35, the default behavior is **the agent writes the code directly in the user's stack** (most muggle-friendly). For each component in the phases 13–15 inventory, produce both:

1. **The spec in `components.yaml`** — props (typed, constraints, required flags), behavior + states, responsive rules at the phase-14 breakpoints (~360/768/1280), accessibility (heading-hierarchy contribution, alt-text, contrast against phase-17 tokens, keyboard order, ARIA the primitive doesn't supply), and `tokens_used`.
2. **Working code** that references phase-17 tokens (never hardcoded values), composes from the chosen library's primitives rather than re-implementing accessible behavior, respects the motion tokens + `prefers-reduced-motion`, and is grouped sanely per stack convention.

Per-stack codegen patterns (Next.js Server/Client boundary, Framer Code Components, WordPress Gutenberg blocks, Astro islands, etc.) → `references/per-stack-codegen.md`.

**Composition rules — enforced, not optional:** one primary library per project; up to two complementary libraries for specialized concerns (motion, data tables); `brand.yaml` tokens drive *all* libraries (mixing a library's own brand preset is forbidden); never double-wrap (Radix directly *or* via shadcn, not both for the same primitive). Compose blocks from primitives (the 2026 shadcn composition model: blocks = meaningful UI sections built from primitives, sitting between primitives and pages) rather than mashing every-shadcn-plus-Aceternity-plus-MagicUI — incoherence is the failure mode the composition cap prevents.

### 5. Self-review sweep — the "read it aloud" cross-check, code edition

Before declaring the component set done, read the whole set against `brand.yaml.tokens` and reconcile drift. This is the canonical phase-18 discipline. Refuse to advance (these gates are not overridable — they are correctness prerequisites for phases 19–23):

- **Any inventory component missing spec + code** — partial coverage is not advance-able; phase 19 cannot compose a page from a component that doesn't exist.
- **A hardcoded value the phase-17 system owns** — a literal hex / font / ad-hoc pixel where a token belongs. Rewrite every one to the token reference.
- **Inconsistent style across the set** — one component on the spacing scale, the next on raw pixels; one keyboard-accessible, the next not; one referencing the motion token, the next hardcoding a duration. Reconcile the drift.
- **Page composition attempted before all components exist** — components first, composition second, non-negotiable.
- **A11y obligations from the spec unmet in the code** — missing alt-text wiring, no keyboard order, missing ARIA, contrast not clearing the phase-17 bar.
- **Component library mixed incoherently** — two primaries; a complementary library's brand preset bleeding in; the same primitive double-wrapped.

Update `project.yaml.current_phase` to `19` only on user confirmation that every component is specced, built, and reviewed.

### 6. JSON handoff protocol — opt-in per component

When the user signals they want an external tool or a freelancer for a given component, do not refuse or lose discipline — emit a structured brief. Surface per component:

```
Building <Component>. Options:
  [1] I write the code directly in your stack          (default; fastest)
  [2] I emit a brief; you take it to your AI tool       (ChatGPT / Claude.ai / v0 / Cursor / Lovable)
  [3] I emit a brief; you send it to your freelancer
  [4] You write it yourself; hand me the result when done
```

Most muggles pick `[1]`. The full round-trip (Flow A out → external tool → Flow B in via phase 6.5, the `component-request-v1` brief shape, per-tool adapter quirks, the human-freelancer fixture, palette-validator drift handling) → `references/json-handoff-protocol.md`. The discipline holds across the tool boundary because the brief carries the brand tokens, voice, and a11y obligations; without it, a muggle pasting "make me a hero" into ChatGPT gets drift.

## 3rd-party plugin composables to recommend (do NOT vendor — point only)

When the user reaches phase 18, recommend (via the `Skill` tool) where the purpose fits — these are the user's to invoke, never bundled or embedded:

- **`document-skills:frontend-design`** — when the user wants distinctive, production-grade visual polish on a component beyond the chosen library's defaults. Recommend before building hero / showcase / landing components.
- **`accessibility-compliance:wcag-audit-patterns`** — when a component carries non-trivial a11y obligations (interactive widgets, forms, modals). Recommend running it against the built component before the phase-18 self-review sweep, so the formal phase-21 audit finds less.
- **`ui-ux-pro-max@ui-ux-pro-max-skill`** — the loaded design-skill flavor's component element library (button/modal/navbar/card/table/form/chart with style-aware variants) as a secondary consistency reference against the phase-17 aesthetic.
- **`document-skills:web-artifacts-builder`** — when the user wants a multi-component interactive artifact (React + Tailwind + shadcn/ui) prototyped quickly before committing it into the project tree.

Recommend; let the user invoke. The skill points, it does not pull.
