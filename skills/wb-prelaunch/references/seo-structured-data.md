## The two disciplines (the gates this phase exists to enforce)

1. **Per-page specificity.** Every page gets a title describing *that page*, a meta description for *that page's* intent and SERP snippet, an OG image/card for *that page*, and schema.org of the type *that page actually is*. **No two pages share a title or a description** — the single-template-everywhere state is the defining failure and is refused (overridable only for genuinely-equivalent pages like paginated archives, with rel-canonical handled and logged).
2. **Truthful structured data.** Markup mirrors visible content, never invents it. `AggregateRating` with no real reviews, an `Offer` price the page doesn't show, `Event` data contradicting the visible date — Google treats markup-not-matching-content as spam and can issue a manual action that tanks the whole domain. **Not overridable.** Invalid JSON-LD (syntax error / missing required property) is also refused — validate and fix before marking the page done.

## Per-page metadata checklist

For every page in `.website-builder/sitemap.yaml`:

- **`<title>`** — unique, describes the page, front-loads the distinguishing term, within practical SERP-display length, brand at the end where it aids recognition.
- **Meta description** — unique, written for the SERP snippet + the page's actual intent, in the phase-5 brand voice (`brand.yaml.voice`), no SEO-keyword soup. (Google may rewrite it; a specific human-written one is still the right input and fallback.)
- **Open Graph** — `og:title`, `og:description`, `og:image` (page-appropriate, correctly sized), `og:type`, `og:url`.
- **Twitter/X card** — `twitter:card` (`summary` or `summary_large_image`) + matching fields.
- **Canonical** — `rel=canonical` correct (esp. for paginated/duplicate-ish pages).

A titleless or description-less page is a broken SERP entry — missing title/description is **not overridable**.

## schema.org type-per-page catalog (JSON-LD — Google's recommended format)

Emit the type the page actually is, in a `<script type="application/ld+json">` block, with the type's core properties populated **from real content**. Prefer the more specific type (`BlogPosting` > `Article`; `LocalBusiness` and its precise sub-type like `LegalService`/`Plumber` > `Organization` for a business with a physical location).

| Page kind | schema.org type | Core properties (from real content) |
|---|---|---|
| Business / local-business home + about | `Organization` / `LocalBusiness` (+ sub-type) | `name`, `url`, `logo`, contact; LocalBusiness adds `address`, `telephone`, `openingHoursSpecification` (+ `specialOpeningHoursSpecification` for holidays), `geo`, precise `areaServed` |
| Site/page baseline | `WebSite` (+ `SearchAction` where site-search exists) + `WebPage` | `name`, `url`, `potentialAction` for search |
| Blog / essay / news | `Article` / `BlogPosting` | `headline`, `author`, `datePublished`, `image`, `articleBody` |
| Transactional product/service | `Product` / `Service` + `Offer` | `name`, `description`, `offers` (`price`, `priceCurrency`, `availability`). `AggregateRating`/`Review` **only if real reviews exist** |
| Event pages | `Event` | `name`, `startDate`, `location`, `offers` |
| Personal-brand / portfolio | `Person` | `name`, `jobTitle`, `sameAs` (social profiles) |
| Deep-site navigation | `BreadcrumbList` | the navigational hierarchy |
| Genuine Q&A pages only | `FAQPage` | only where content is genuinely Q&A **and** the rich result is still attainable — Google narrowed FAQ/HowTo eligibility; confirm *current* eligibility via the fetched Google guidance rather than blanket-adding FAQ markup chasing a rich result Google no longer shows |

2026 best-practice (confirmed via WebSearch 2026-05-18): the foundational trio is `Organization` (or `LocalBusiness`), `WebSite`, `BreadcrumbList` on every page; then page-specific types; expand only when a specific rich-result opportunity matches the content. A dozen accurate validated blocks beat fifty half-implemented ones.

## Crawl-control files (build-wired, not hand-frozen)

- **sitemap.xml** — every indexable page, **generated by a build step so it regenerates on content change** (a one-time hand-frozen sitemap is a phase-29 + post-launch drift source — refused). Excludes noindex/thin pages.
- **robots.txt** — allow the public site, disallow admin/preview paths, point at the sitemap. Verify it is the *production* file: a `Disallow: /` left from a staging config silently deindexes the whole site (a parity failure analogous to phase 24's) — refused.

## Validation workflow (before marking each page done)

1. Run the page's JSON-LD through the **Google Rich Results Test** (Google-specific rich-result eligibility) and the **schema.org Markup Validator** (generic schema validity) — via `WebFetch`/`Bash` against the validator endpoints or public tools.
2. Fix syntax errors / missing required properties rather than shipping markup that does nothing.
3. **Server-render check** — confirm the metadata + JSON-LD is in the *server-rendered* `<head>` (Playwright against view-source / SSR output). Client-only metadata set via JS after hydration is invisible to many crawlers and LLM fetchers — a common SPA failure.

## The LLM-discoverability check (2026)

LLMs are now a primary brand interpreter — conversational search reads the page's metadata + structured data to represent the brand. For each page, verify the title/description/structured-data reads coherently *to a model*, not just a crawler: the positioning paragraph (phase 3) and the structured `Organization`/`Person` data are what an LLM surfaces when a user asks it about the brand. Note this per page in the report.

## Stack-specific embedding

Embed per the stack's mechanism: Next.js Metadata API / `generateMetadata`; Astro `<head>` component; WordPress SEO via theme or a plugin's structured-data; Framer page settings. Wire sitemap.xml + robots.txt into the build.
